# feature — 신규 제품기능 (T3)

새 기능은 "무엇을 왜"가 먼저다. 코드부터 짜면 이미 있는 걸 다시 만들거나(중복 제안) 스코프가 새어 나간다.
이 스킬은 **정의 → 분해 → 병렬 구현 → 검증 → 배포** 파이프라인을 못 박는다.

> ⚠ **트리아지**: 스코프가 명확하고 1~2파일이면 T3가 아니라 T1(직접편집+verify)이다. 여러 레이어를 새로 만들거나
> "이 기능 어때"처럼 스코프가 열려 있으면 T3. 애매하면 한 티어 위로(더 신중히).

## 트리거
"이 기능 추가해줘 / 어때 / 새 화면 / 새 도메인" + 스코프·수용기준이 아직 안 잡힘.

## 절차

### 1. 정의 — pm-po (문제·목표·AC·비목표)
- `pm-po`가 **문제·목표·성공지표·유저스토리·수용기준(AC)·비(非)목표**를 뽑는다. 비목표를 명시해야 스코프가 안 샌다.
- **제안/설계-전 단계면 read-only 병렬 교차검증**을 붙인다: `pm-po` ∥ `ui-ux-designer`(+데이터가 관건이면 `ticker-data-curator`)를
  동시에 read-only로 띄워 "현재 코드 사실관계"를 교차확인한다. 오케스트레이터 브리핑의 사실조차 낡을 수 있다(실측:
  "진행률 게이지 없음"으로 브리핑했으나 이미 `StatTile` 진행바가 있었고 target=0 버그까지 잡힌 사례). 각 에이전트에
  "`npm run search`로 사실 교차확인"을 명시한다.
- 파괴적/전환영향 결정(기본값·기능제거·스키마·타사 브랜드노출·파괴적 플로우)은 **독단 금지 → 사용자 확인**.

### 2. 분해 — orchestrator
- `orchestrator`가 독립 하위작업으로 나눈다. **병렬 트랙은 파일 경계로 분리**하고, 공유 파일(`copy.ts`, `index.ts` 배럴,
  `analytics.ts` 택소노미)은 순차 처리하거나 담당을 지정한다("누가 어떤 파일을 만지는지"를 착수 전에 명시).
- **계약(어트리뷰트명·저장 키·함수명)은 먼저 랜딩한 구현이 정본** — 스펙 문서가 다르게 쓰면 취합자가 즉시 조정.

### 3. 구현 — specialist 병렬
- 도메인별 스페셜리스트에 위임: `simulation-engineer`(엔진) · `state-engineer`(jotai·영속·공유) ·
  `frontend-engineer`(화면) · `ui-ux-designer`(흐름·반응형·접근성·카피) · `ticker-data-curator`(데이터).
- 🔴 **브리핑에 지식을 주입한다 — 서브에이전트는 이 대화를 못 본다.** 관련 `.claude/knowledge/`의 **확정 결정·함정·제약**을
  발췌해 브리핑에 인라인한다(실측 효과: pitfalls의 "vi.mock 전체치환 목 누락 크래시"를 주입하니 frontend가 선제 차단 →
  전 LoginModal 테스트 붕괴 예방). **확정 결정은 "이미 선택됨" 전제로 못 박아** 재설계 왕복을 없앤다.
- **동일 도메인 후속은 이전 작업자를 재개**(SendMessage)하는 편이 새 에이전트보다 빠르고 정확했다(컨텍스트 유지).
- ⚠ **지식파일 편집은 orchestrator 전담**으로 브리핑에 명시 — 서브에이전트가 pitfalls를 먼저 고치면 큐레이션과 편집 충돌이 난다.

### 4. 검증 — qa + reviewer + verify
- `qa-tester`: **사용자 행동 기반** 테스트(className/Emotion 내부구현 기반 금지). 훅 IO를 직접 단정해야 하는 것(수정모드 값 뒤집힘 등)은 뷰 테스트로 대체 못 한다.
- `reviewer`: 머지 전 **정확성·구조규칙·하위호환·데이터안전**만 본다(수정 안 함). "blocking만, 없으면 통과"로 게이팅하면 대기 없이 완주.
- `npm run verify`로 tsc→test→api→build 그린 확인. **계산·저장·공유 링크를 건드렸으면 왕복 테스트 필수**(CLAUDE.md "가장 조심할 것").
- ⚠ ui-ux와 pm-po가 **상충 권고**(예: "지금 배포" vs "측정 먼저")를 내면 오케스트레이터가 성급히 봉합하지 말고 **상충을 그대로 사용자에게** 넘긴다.

### 5. 배포 — ship
검증 그린이면 `ship` 스킬(T4). ⚠ [[ask-before-deploy]] 매번 승인.

## 검증(요약)
AC 충족 + qa 그린 + reviewer blocking 0 + `npm run verify` 그린. 핸드오프엔 **검증 로그 원문 수치**(테스트 개수·대비 값)를 담는다("통과했다"만 쓰지 말 것).

## 함정 압축
- 브리핑 사실관계 낡음 → read-only 병렬 교차검증. · 서브에이전트는 대화 못 봄 → 지식 주입. · 공유 파일 병렬 충돌 → 파일 경계 분리. · 지식파일 이중편집 → orchestrator 전담.

## 진화
이 절차가 현실과 어긋나거나 더 나은 길이 보이면 `.claude/knowledge/retro.md`에 근거를 남기고 이 스킬을 고쳐라. 프로세스는 살아있다(dev-process 마스터 §4).
