## Git — 선택적 SHIP/HOLD 배포 (부분 스테이징 커밋·격리 빌드)
- [2026-07-23][git] **부분 스테이징 파일은 `git commit -- <pathspec>`으로 커밋하면 안 된다** — pathspec을 주는 커밋의 기본 동작은 `--only`(그 경로의 "작업트리 현재 내용"을 그대로 커밋, index staged 여부 무관)라서 `git add -p`로 일부 훵크만 올리고 나머지를 unstaged로 남긴 파일(예: analytics.ts의 GA4가드 vs reconcile 택소노미)을 pathspec 커밋하면 unstaged 훵크까지 같이 실린다. 해법: 다른 그룹을 전부 pathspec 커밋으로 먼저 떼어내 인덱스에 그 파일 하나만(staged 부분만) 남긴 뒤, **pathspec 없는 `git commit`**(순수 인덱스 커밋)으로 마무리한다.
- [2026-07-23][git] **HOLD 코드를 작업트리에 남긴 채 격리 빌드를 검증하려면 `git stash push -u --keep-index` → 빌드 → `git stash pop`이 아니라 `apply --index`로 먼저 복원 검증 후 `drop`** — plain `stash pop`(옵션 없음)은 staged/unstaged 구분을 복원 못 해 부분 스테이징이 통째로 unstage된 채 돌아온다. 같은 이유로 **SHIP 커밋 완료 후 다른 브랜치로 체크아웃할 때도** SHIP 커밋에 포함된 파일이 HOLD uncommitted 잔여도 갖고 있으면(analytics.ts처럼 훅 일부만 커밋) "local changes would be overwritten"로 checkout이 스스로 거부한다(안전, 데이터 손실 아님) — HOLD 전체를 다시 `stash push -u`로 대피 후 전환, 대상 브랜치에서 `apply` → diff 대조 → `drop`.
- [2026-07-23][git] 세션이 길어져 로컬 main이 origin보다 여러 커밋 뒤처지면, 여러 세션이 공유 append하는 지식문서(decisions.md·pitfalls.md)가 upstream과 겹칠 수 있다 — push/PR 전에 `git diff --stat <로컬HEAD> origin/main`로 SHIP 파일과의 교집합을, `git merge-tree --write-tree <branch> origin/main`(읽기전용·작업트리 불변, exit 0+tree OID만 출력=클린)로 실제 충돌 여부를 미리 확인해두면 병합 단계에서 놀라지 않는다.
- [2026-07-23][git] "리컨사일 코드 누출 없음" grep 검증은 지식문서(사용자가 "통째로 SHIP 허용"한 decisions.md 등)에 리컨사일 키워드가 매치되는 것과 실제 코드(jotai/snowball/cloud/* 등)에 매치되는 것을 **구분해서** 봐야 한다 — 전자는 팀 지식(정상), 후자만 진짜 누출이다. 무차별 전체 grep 결과만 보면 사용자가 명시적으로 허용한 문서 항목을 "스테이징 실수"로 오판한다.
