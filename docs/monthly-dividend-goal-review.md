## 0. 한 줄 요약

**조건부 GO.** 만들 가치는 분명하지만 **스펙 원문 그대로가 아니라 스코프를 좁혀서**. 스펙 인프라의 절반이 이미 있는 것과 중복이거나 알려진 병목(지급일 캘린더·환율)을 밟는다. 핵심 신가치는 "도달 예상일 계산"이 아니라 **입력 모델의 전환**(비중%+가정률 → 실보유 수량 × 실제 주당배당)이다.

## 1. 전제를 뒤집은 발견 3가지

검토 중 스펙의 핵심 전제 3개가 실측/코드로 반증되었다.

1. **Supabase Edge Function / Cron은 불필요하다.**
   데이터 소스로 쓸 Yahoo는 무키(無key) API라 "API 키 보호" 명분이 약하다. 게다가 이미 **Vercel 서버 함수(`api/*.js`, `server/handlers/*`) + `tickerRefresh` 월간 크론(GitHub Actions)** 이 돌고 있다. 장기 예측에서 가격·배당은 일 단위로 유의미하게 안 움직이므로 **기존 파이프라인 확장**으로 충분하다. → 신규 인프라 도입 회피.

2. **Yahoo가 한국 상장 ETF도 커버한다.**
   `458730.KS`(TIGER 미국배당다우존스) 직접 호출로 **현재가 + 분배금 이력 23건** 정상 반환 확인. 기존 미국용 파이프라인에 `.KS`/`.KQ` 접미사만 붙이면 **KR + US를 단일 소스로** 처리 가능. → "한국 데이터가 병목"이라는 전제 완화.

3. **"목표 달성 예상일 계산"은 새 가치가 아니다.**
   기존 계산 엔진에 이미 있다(`findTargetYear`). **진짜 새로운 것은 입력 모델** — 추상 가정값이 아니라 실보유 수량 × 실제 주당배당.

## 3. 계산 로직 검토

### 재사용 (그대로 호출)
- 세금 계산(`computeMonthlyPayout`, 배당 15.4% 세후)
- 재투자 타이밍(`planReinvestment`, same/next-month)
- 달력 유틸(`toStartDate`/`addMonths`, 말일 클램프)
- zod null-반환 검증 패턴
- 포트폴리오 Σ 합산 골격

### 신규 필요 (코드베이스 전무)
- 실보유 **수량** 입력 (엔진은 `가격 × 배당률%`로 DPS를 역산 — 절대통화 실DPS 개념 없음)
- **월 해상도 도달일** (`findTargetMonth`) — 기존은 연 해상도
- 정수주 추가매수 + 잔액 이월(carry) 버킷
- 실지급 캘린더

### 접합 판정
`runSimulation` **본체는 파라미터만 바꿔서 재사용 불가** — 3가지 이유:
1. 소수주 고정 (실보유는 정수주 현실)
2. 지급월이 *시작월 상대* (실제 SCHD는 *달력 고정* 3/6/9/12월)
3. 목표 탐색이 연 해상도

→ **별도 순수함수 `runHoldingsProjection` 신설 + 하위 헬퍼 공유**가 맞다.

### ⚠️ 반드시 결정할 계산 함정
| 함정 | 내용 |
|---|---|
| **달성률 기준** | "연배당/12 평균"(엔진 정합) vs "실지급월 배당"(캘린더 정합). 분기/반기 ETF는 특정 달만 배당 → 둘이 반드시 어긋남 |
| **지급월 off-by-one** | 기존 `isPayoutMonth`는 *시작월 기준 상대*(2월 매수→5월 첫 지급). 실캘린더(달력 고정) 원하면 재사용 불가 |
| **성장 반영** | 스펙 "현 데이터 유지" = growth 0 고정. 실DPS 재현하려면 `배당률 = 실연배당/현재가` 역산 브리지 필요 |
| **정수주 vs 소수주** | 정수주면 carry 버킷 신규 필수(안 하면 잔액 증발 → 자산 하향편향). 소수주면 엔진 재사용↑·국내 브로커 현실성↓ |
| **세전/세후** | 기존 앱은 전면 세후. 스펙은 세금 무언급 → 세전으로 읽힘. 목표 월배당이 세전/세후 어느 쪽인지 명시 안 하면 기존 화면과 숫자 불일치 |

## 5. 아키텍처

### 테이블 — 2종류로 갈림
- **사용자 소유** (`user_goal`, `user_holdings`): 기존 `user_app_states` 템플릿 그대로.
  - FK → profiles CASCADE, `default auth.uid()`
  - 컬럼 GRANT에서 `user_id` 제외 = 위조 불가
  - owner-only RLS 4정책, 쿼터 트리거
  - ⚠️ **`user_id` 컬럼은 스펙에 빠졌지만 필수**
- **공통 데이터** (`etf`, `dividend_schedule`): 선례 없는 패턴.
  - `grant select to anon, authenticated` + RLS `using(true)` (비로그인 계산 위해)
  - **클라이언트 write GRANT 절대 없음** (service_role 크론만 write)
  - `etf.updated_at`은 **UI에 신선도 라벨로 반드시 노출** (무근거 표시·무음 실패 방지)

### 저장 방식
정규화 테이블 채택 (payload JSON 슬롯 아님) — "실보유 CRUD + 시세 조인" 성격. IO는 `shared/lib/supabase/`에 `userGoals.ts`·`userHoldings.ts`·`etf.ts` (기존 `userAppStates.ts` 복제, 배럴 미연결 + 폴더경로 소비).

### API 호출 위치
Supabase Edge Function 신규 도입 불필요 → **기존 `tickerRefresh` 파이프라인 확장** (배당 히스토리 persist 추가만).

> ⚠️ 주의: 시뮬레이터는 **생성물 JSON에서 가격을 읽는다.** 추적기가 DB `etf`를 새 정본으로 두면 **가격 출처가 이중화**된다. `dividend_schedule`처럼 JSON에 없는 데이터만 DB로 가는 게 중복 최소.

### 하위호환
**도메인 분리 시 자동 성립.** 기존 저장 페이로드·공유 URL(v1/v2/v3)·IndexedDB를 **한 줄도 안 건드리고** 추적기를 별개 도메인으로 얹으면 리스크 0.
→ **실보유 수량·종목은 공유 URL(lz-string)에 넣지 말 것** (사적 데이터 유출 + URL 폭증).

### 환율
- `etf.currency` = 네이티브 통화 정본
- `user_goal.currency` = 사용자 표시 통화 (역할이 다름)
- #11(실시간 환율) 없이는 원화 + 달러 혼합 배당을 한 통화로 합산 불가 → **블렌드 숫자 지어내지 말 것.**
- MVP는 (a) 단일 통화 추적기 or (b) 통화별 소계만. 지금부터 네이티브 통화로 저장해 두면 #11이 무마찰 슬롯인.
